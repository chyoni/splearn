## 컨트롤러, 서비스, 도메인 계층 모두 다 데이터 검증

회원 생성 시 컨트롤러 레이어에서, 애플리케이션 레이어에서 받는 파라미터에 `@Valid`, `@Validated`로 검증을 해도,
도메인에서도 데이터 검증을 했다. 이는 매우 당연한 사실이지만 중요한 부분이다.

회원 생성을 처리하는 경로가 반드시 컨트롤러를 통해, 애플리케이션 레이어를 통해서만 처리되지 않을 수 있기 때문이다.
배치로 돌리는 회원 생성의 경우, 아마 도메인에 직접 접근할 수 있을 것이다.

따라서, 모든 레이어에서 검증할 수 있으면 좋다. 검증하기도 쉽고!

## 애플리케이션에서 DTO로 반환해서 넘겨야한다는 편견을 버리자

도메인 클래스 자체를 넘겨도 아무런 문제도 없고, 이게 사실 의존성의 방향으로 따지면 더 올바른 방향이다.

웹 레이어가 원하는 형태의 데이터로 애플리케이션에서 변환하는 것은 애플리케이션이 어댑터를 의존하는 잘못된 의존 방향을 가지는 것이다.

어댑터는 도메인 클래스를 받고 자신이 원하는 형태로 변환하여 반환하면 된다.

OSIV (Open Session In View)는 나쁜게 아니다. 오히려 lazy 로딩을 더 수월하고 편하게 도와주는 도구에 가깝다.

OSIV 켜져 있고, 도메인이 어댑터까지 넘어가게 되면 어댑터에서 도메인 변경할 수 있는거 아니냐고 하는데 그렇지 않다.

트랜잭션이 애플리케이션에서 종료되면 OSIV가 켜져 있더라도 변경 작업을 할 수 없다. 다만 조회가 가능할 뿐이다.

서비스가 너무 커져버린 대성공한 회사의 제품이라면 OSIV가 성능 영향을 줄 수 있겠지만 그렇지 않다면 크게 이것을 부정하지 않아도 된다
